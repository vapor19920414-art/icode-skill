# 步骤 1 — 拟定正式项目计划

**命令**: `/icode plan <需求> [--focus <路径>...]` 或 `/icode new <需求> [--focus <路径>...]`
**产出**: `{ICODE_OUT_DIR}/01_plan.md`
**会话**: 主会话

在**当前会话**中直接撰写计划，保留所有历史对话中的需求上下文。

## 执行步骤

1. 执行目录管理中的「创建新目录」逻辑，确定 `ICODE_OUT_DIR`

2. **解析 `--focus` 参数**（步骤 0，最先执行）：
   - 从调用参数中检测 `--focus` 标记
   - `--focus` 之后的空格分隔路径列表即为 **focus_paths**（相对于项目根目录）
   - 若未提供 `--focus`，focus_paths 为空
   - 示例：`/icode plan 新增蓝牙接口 --focus broker/src/op_mcu_communication.cpp product_test/product_test_node/`
     → focus_paths = [`broker/src/op_mcu_communication.cpp`, `product_test/product_test_node/`]

3. **聚焦代码探索**（替代原有的"先了解现有工程"）：
   
   核心原则：**绝不 `find . -name "*.cpp"` 全盘扫项目**。token 消耗必须控制在合理范围。

   ### 3a. 有 `--focus` 时（精准模式）

   **探索范围 = focus_paths + 直接依赖**：
   
   1. 用 Read 工具逐一阅读 focus_paths 中的文件
   2. 从这些文件中提取 `#include` 引用的**项目内部头文件**（非系统头文件），也纳入阅读
   3. 从代码中识别关键函数调用，追踪到被调用方所在文件（仅限项目内部、直接调用的核心文件）
   4. **禁止**：从 focus_paths 出发递归扫描整个子目录树、读取与需求无关的文件
   
   探索深度：**最多 2 层依赖**（focus 文件 → 其直接 include/调用的文件），总文件数建议 ≤ 15 个。

   ### 3b. 无 `--focus` 时（关键词驱动模式）

   从用户需求中提取 **2-5 个核心关键词**（函数名、类名、文件名、模块名），用 grep/rg 精准搜索：
   
   ```bash
   # 示例：需求是"新增蓝牙开启接口"
   rg -l "bluetooth\|Bluetooth\|BT_\|hciconfig" --type cpp --type c --type h
   rg -l "EnableBluetooth\|enable_bluetooth\|BT_ENABLE" --type cpp --type c --type h
   ```
   
   只阅读 grep 命中的文件（取 Top-N 相关度最高的，≤ 10 个）。**绝对禁止**无关键词的 `find`/`rg` 全量文件列表扫描。

   ### 3c. 探索产出（写入计划）

   无论哪种模式，探索完成后在计划的「架构设计 → 跨文件关联分析」中记录：
   - 已探索的文件列表（路径 + 为何相关）
   - 识别到的可复用模块/函数/模式
   - 识别到的关键约束（命名规范、错误处理风格、接口签名约定）

4. **深度思考**（必须先执行，不可跳过）：需求分解 → 方案分析 → 风险评估
   - 基于步骤 3 的探索结果（而非凭空设想）做方案分析
   - 如果探索后发现 focus_paths 不足以覆盖需求，在深度思考中明确指出还需要了解哪些文件

5. 撰写计划：
   包含以下 9 个章节（缺一不可）：

需求描述（包含所有对话中讨论过的细节）：
{用户输入的原始需求}

必须包含的章节（逐一输出，不得跳过）：
1. **项目概述** — 目标、范围、约束条件，需说明与现有工程的关系
2. **功能需求** — 所有功能点列表，含输入/输出/边界，标注哪些可复用现有模块
3. **架构设计** — 模块划分、数据流、接口定义，需说明如何在现有架构中扩展。**必须包含跨文件关联分析**：哪些文件需要新建、哪些现有文件需要修改、修改的文件被谁依赖。**必须列出步骤 3c 的探索产出**（已探索文件 + 关键发现）
4. **架构决策记录（ADR）** — 每个关键决策记录：上下文、候选选项、决策、理由。格式如下：

   ```markdown
   ### ADR-N: {决策标题}
   - **上下文**：为什么需要做此决策
   - **选项**：A) xxx  B) xxx  C) xxx
   - **决策**：选择 X
   - **理由**：为什么选 X 而非其他
   ```

   需记录 ADR 的典型场景：方案选型、降级策略、接口取舍、兼容性权衡。决策变更时只需更新此章节 + 受影响的引用点。
5. **模块详细设计** — 每个模块的职责、关键函数、数据结构，引用现有接口命名风格。**代码示例使用伪代码+关键行号引用**，禁止粘贴完整函数实现（完整实现留给步骤4编码阶段）。格式：`参考 src/foo.cpp:42-68 的 HandleXxx 模式`
6. **异常处理** — 错误码、异常场景、降级策略
7. **实现步骤** — 分阶段实施顺序、依赖关系
8. **校验项** — 可复核检查点列表（用于后续步骤核对）
9. **风险评估** — 技术风险、依赖风险、缓解措施

格式要求：
- 使用 Markdown 格式
- 条理分明，每章有编号
- 校验项以 [ ] checkbox 形式列出

6. **断言验证**（必须执行，不可跳过）：
   从计划中提取所有涉及具体代码位置/行为的断言（如"某函数在文件X中"、"某配置从Y加载"、"某接口签名是Z"），逐一用 Read/Grep 工具实证验证。
   - **验证范围约束**：只验证步骤 3 探索范围内的文件，不扩展到未探索区域
   - 已验证的断言标记为 `[已验证]`
   - 验证失败或未验证的断言标记为 `[未验证]`，并说明原因
   - 验证失败导致计划需要调整的，立即修正对应章节
   - 将验证结果追加到计划末尾的"断言验证记录"章节

7. 使用当前会话可用的文件写入工具，将计划写入 `{ICODE_OUT_DIR}/01_plan.md`

## Token 效率红线

- **禁止** `find . -name "*.cpp"` 或 `find . -type f` 全盘文件列表
- **禁止** `rg ".*"` 或等价的全量匹配
- **禁止** 递归列出整个子目录树（如 `tree`、`ls -R`）
- **禁止** 阅读与需求明显无关的模块/文件
- 单次探索读取文件数建议 ≤ 15 个
- 若用户提供了 `--focus` 但范围仍过大（如指定了 10+ 个目录），只取其中最核心的 3-5 个路径探索

## 强制操作（完成后必须执行）

8. **创建 `{ICODE_OUT_DIR}/.ico_metadata.json`**：

```json
{
  "requirement": "{用户输入的原始需求}",
  "created_at": "当前时间",
  "status": "plan_done",
  "completed_steps": ["1"],
  "code_files": [],
  "focus_paths": ["{--focus 传入的路径列表，无则为空数组}"]
}
```

9. 如果是 `/icode new`（全流程模式）：

   - **立即继续执行步骤2**（不要等待用户确认）
   - 如果会话断开后恢复，读取 `.ico_metadata.json` 的 `completed_steps`，从最后一个完成步骤的下一步继续（如 `["1"]` → 步骤2）
